# hccl-dev 缺陷模式归纳

从10条缺陷提交中归纳出5个缺陷类别，均从数据中自然涌现。

## 类别1：枚举/标识符混淆（3条，30%）

当存在语义相近但实际不同的枚举值或标识符时，使用了错误的变体。

| 维度 | 内容 |
|------|------|
| 出现频次 | 3 |
| 典型案例 | 7347fee3 (AICPU vs AICPU_TS, 6处全错), 13b6032d ("AICPU" vs "AI_CPU"), 2d8d548a (blockDim vs numBlocks) |
| 根因 | CANN平台存在大量命名相近的枚举值/字符串常量，开发者容易混淆 |
| 审查检查点 | (1)使用引擎类型枚举时，确认具体变体是否正确(AICPU/AICPU_TS/CPU/CPU_TS/AIV等)；(2)字符串字面量与平台约定是否一致；(3)API参数名变更后是否全局同步 |

## 类别2：接口变更传播不完整（3条，30%）

结构体/API变更后，部分使用方未同步更新，导致编译或链接失败。

| 维度 | 内容 |
|------|------|
| 出现频次 | 3 |
| 典型案例 | c11a5289 (HcclCommConfig字段移除未同步到测试), 8222fcf8 (同+缺少include), beb0ed54 (CMakeLists缺少源文件) |
| 根因 | API处于快速迭代期，修改定义侧时未充分搜索所有引用方(尤其是测试代码和构建配置) |
| 审查检查点 | (1)修改struct/class定义时，grep全部使用点包括test/mock/stub；(2)新增库目标时检查所有被引用的符号是否有对应源文件；(3)跨编译环境(Host/AICPU)的代码需确认API可用性 |

## 类别3：复制粘贴错误（2条，20%）

从相似代码复制后未完整替换变量名或标识符。

| 维度 | 内容 |
|------|------|
| 出现频次 | 2 |
| 典型案例 | 11b7211a (run_st中检查ENABLE_UT而非ENABLE_ST), 8222fcf8 (同一bug的引入提交) |
| 根因 | run_st()从run_ut()复制，但条件变量名未替换 |
| 审查检查点 | (1)新增函数如果结构与已有函数雷同，逐行对比是否所有标识符已替换；(2)构建脚本中的开关变量(ENABLE_XX)与对应的命令行参数(--xx)是否匹配 |

## 类别4：条件判断逻辑不完整（1条，10%）

分支条件遗漏了必要的组合因素。

| 维度 | 内容 |
|------|------|
| 出现频次 | 1 |
| 典型案例 | 19d20206 (仅检查设备类型忽略运行模式环境变量) |
| 根因 | 设备类型与运行模式的组合约束未在代码中完整表达 |
| 审查检查点 | (1)设备类型分支是否考虑了运行模式/环境变量等组合条件；(2)新增设备类型时是否需要更新所有判断点 |

## 类别5：CMake/构建配置错误（1条，10%）

CMake脚本中的版本兼容性和语法格式问题。

| 维度 | 内容 |
|------|------|
| 出现频次 | 1 |
| 典型案例 | 1f13573a (CMAKE_CURRENT_FUNCTION_LIST_DIR版本兼容性 + -D空格问题) |
| 根因 | CMake的版本特性差异和参数格式陷阱 |
| 审查检查点 | (1)项目最低CMake版本要求与使用的特性是否匹配；(2)-D参数后不加空格 |

## 附：API函数命名错误

cae52923 (HcclBatchSendRecvInner→HcclBatchSendRecv) 可归入"接口变更传播不完整"的子类 — 公共API函数名与头文件声明不匹配。

## 跨类别观察

1. scatter_op.cc集中了类别1和类别4的缺陷，是该仓库的核心风险文件
2. 类别1和类别2合计占60%，均与API快速迭代相关 — 这是开发期仓库的典型特征
3. 与hccl主仓相比，hccl-dev的缺陷更多是"接口不匹配"类而非"逻辑错误"类，反映了开发分支侧重API设计/重构的特点
4. Revert分析进一步印证了API不稳定：API重命名(MR !34)同日被revert，打包路径结构(MR !18)2天内经历3次变更
